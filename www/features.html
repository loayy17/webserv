1. Overview & Architecture

  • Language: C++98 (-std=c++98 -Wall -Wextra -Werror)
  • I/O Multiplexing: Single poll() for ALL I/O (sockets, pipes, listen)
  • Non-blocking: All sockets & CGI pipes set to O_NONBLOCK via fcntl()
  • No read/write without poll(): Every I/O goes through poll readiness check first
  • No errno checking after read/write: Behavior based on return values only
  • CGI: Only uses fork() for CGI; the rest is event-driven
  • Keep-Alive: HTTP/1.1 persistent connections with proper pipelining
  • Timeouts: Client timeout (60s) and CGI timeout (60s)

2. Configuration File Guide

The configuration follows an NGINX-like syntax. Provide it as argument: ./webserv webserv.conf

Minimal Configuration

server {
    listen 0.0.0.0:8080;
    server_name localhost;
    root ./www;
    index index.html;
}

Full Configuration Example

http {
    client_max_body_size 5M;       # Global body size limit

    server {
        listen 0.0.0.0:8080;        # IP:port to bind
        server_name localhost;       # Server name (virtual host matching)
        root ./www;                  # Default document root
        index welcome.html;         # Default index file

        # Custom error pages
        error_page 404 ./www/errors/404.html;
        error_page 500 ./www/errors/50x.html;

        # Static files
        location / {
            root ./www;
            index welcome.html;
            methods GET POST DELETE;
            autoindex on;              # Enable directory listing
            client_max_body_size 5M;   # Per-location body limit
        }

        # File uploads
        location /upload {
            root ./www;
            methods GET POST;
            upload_dir ./uploads;      # Where uploaded files are stored
            client_max_body_size 10M;
        }

        # Directory listing
        location /subdir {
            root ./www/subdir;
            methods GET;
            autoindex on;
        }

        # CGI — Python
        location /cgi-bin {
            root ./www/cgi-bin;
            methods GET POST;
            cgi_pass .py /usr/bin/python3;   # extension → interpreter
        }

        # HTTP redirect
        location /old-page {
            return 301 /;             # 301 redirect to /
        }

        # DELETE files
        location /delete {
            root ./uploads;
            methods GET DELETE;
            autoindex on;
        }
    }

    # Second server on different port
    server {
        listen 0.0.0.0:8081;
        server_name second.localhost;
        root ./www;
        index welcome.html;
        client_max_body_size 1M;

        location / {
            root ./www;
            index welcome.html;
            methods GET;
        }
    }
}

3. All Directives Reference

Directive Context Description Example
listen server IP:port to bind to listen 0.0.0.0:8080;
server_name server Hostname for virtual host matching server_name example.com;
root server, location Document root directory root ./www;
index server, location Default file(s) for directory requests index index.html index.htm;
error_page server Custom error page path error_page 404 ./www/errors/404.html;
client_max_body_size http, server, location Max request body size (B/K/M/G) client_max_body_size 10M;
methods location Allowed HTTP methods methods GET POST DELETE;
autoindex location Enable/disable directory listing autoindex on;
upload_dir location Directory for uploaded files upload_dir ./uploads;
cgi_pass location Map file extension to CGI interpreter cgi_pass .py /usr/bin/python3;

Suffix Meaning Example
(none) Bytes client_max_body_size 1024; = 1024 bytes

M / m Megabytes client_max_body_size 10M; = 10 MB
G / g Gigabytes client_max_body_size 1G; = 1 GB

4. HTTP Methods

GET Retrieve resources (static files, directory listings, CGI output)

POST Submit data (file uploads, form data, CGI input)

DELETE Remove files from the server

PUT Upload/replace resources (also supports file upload)

Test with curl

# DELETE a file

5. Static File Serving

The server resolves file paths by combining root + URI remainder after location match:

location /subdir {
    root ./www/subdir;
    index file.html;
    methods GET;
}

Request GET /subdir → serves ./www/subdir/file.html

Request GET /subdir/file.html → serves ./www/subdir/file.html

MIME types are auto-detected from file extension (e.g. .html → text/html, .css → text/css).

7. File Upload

Configuration

location /upload {
    root ./www;
    methods GET POST;
    upload_dir ./uploads;          # Files stored here
    client_max_body_size 10M;      # Max upload size
}

Supported formats:

  • multipart/form-data — Standard HTML form upload (filename extracted from Content-Disposition)
  • Raw body — Direct POST with Content-Disposition header or auto-generated filename

Try Upload


curl examples

# Upload with multipart/form-data
curl -X POST -F "file=@photo.jpg" http://localhost:8080/upload

# Upload with raw body
curl -X POST --data-binary @document.pdf \
  -H "Content-Type: application/octet-stream" \
  -H "Content-Disposition: attachment; filename=\"document.pdf\"" \
  http://localhost:8080/upload

10. Error Pages

Custom error pages can be set per server block. If not configured, a styled default error page is generated automatically.

error_page 404 ./www/errors/404.html;
error_page 500 ./www/errors/50x.html;

Features:

  • Animated styled default error pages for any HTTP status code
  • Custom error page files override defaults when configured
  • Correct HTTP status codes in responses
  • 405 Method Not Allowed includes Allow header with permitted methods
Trigger 404 Trigger 403 
# 404 Not Found
curl -v http://localhost:8080/this-does-not-exist

# 405 Method Not Allowed (DELETE on GET-only location)
curl -X DELETE http://localhost:8080/subdir/

# 413 Payload Too Large
curl -X POST -d "$(python3 -c "print('A'*20000000)")" http://localhost:8080/upload

11. CGI Execution

location /cgi-bin {
    root ./www/cgi-bin;
    methods GET POST;
    cgi_pass .py /usr/bin/python3;    # Python CGI
}

How it works:

  1. Server receives request matching CGI location
  2. fork() creates child process
  3. Child: dup2() pipes stdin/stdout → execve() interpreter with script
  4. Parent: streams body to CGI via pipe (supports chunked + large bodies)
  5. CGI output parsed: Status header, Content-Type, Set-Cookie, body

Environment variables passed to CGI:

Variable Description
REQUEST_METHOD GET, POST, etc.
QUERY_STRING URL query parameters
CONTENT_TYPE Body content type
CONTENT_LENGTH Body size in bytes
SCRIPT_NAME CGI script URI path
PATH_INFO Extra path after script
SERVER_NAME Server hostname
SERVER_PORT Server port number
GATEWAY_INTERFACE CGI/1.1
HTTP_* All HTTP request headers

Try CGI — Database API

# GET — List all users
curl http://localhost:8080/cgi-bin/database.py

# POST — Create user (JSON)
curl -X POST -H "Content-Type: application/json" \
  -d '{"username":"alice","password":"secret"}' \
  http://localhost:8080/cgi-bin/database.py

# POST — Create user (form-urlencoded)
curl -X POST -d "username=bob&password=pass123" \
  http://localhost:8080/cgi-bin/database.py

12. Cookies & Session Management BONUS

How Sessions Work:

  1. Client sends login request to CGI script
  2. CGI returns Set-Cookie: webserv_sid=<32-char-hex>
  3. Server-side SessionManager stores session data (user ID, creation time)
  4. On subsequent requests, cookie is sent automatically → session validated
  5. Sessions expire after 30 minutes of inactivity
  6. Expired sessions automatically cleaned up every 60 seconds

Implementation details:

  • Session IDs: 32-character hex string from /dev/urandom
  • Cookie name: webserv_sid
  • Cookie parsing: RFC-compliant, supports multiple cookies
  • Session timeout: 1800 seconds (30 minutes)
  • Cleanup interval: 60 seconds

Try Session

Go to Login Page Go to SignUp Page 
# Login and receive session cookie
curl -v -X POST -d "username=testuser&password=pass" \
  http://localhost:8080/cgi-bin/database.py
# Response includes: Set-Cookie: webserv_sid=a1b2c3d4...

# Use session cookie in subsequent requests
curl -v -b "webserv_sid=a1b2c3d4..." http://localhost:8080/cgi-bin/database.py

13. Multiple CGI Types BONUS

The server supports multiple CGI interpreters per location, mapped by file extension: 

location /cgi-bin {
    root ./www/cgi-bin;
    methods GET POST;
    cgi_pass .py /usr/bin/python3;      # Python scripts
    cgi_pass .sh /bin/bash;             # Bash scripts
    cgi_pass .pl /usr/bin/perl;         # Perl scripts
}

Each extension is independently mapped to its interpreter. The server detects the extension from the request URI and uses the correct interpreter.

The cgi_tester provided by the subject uses .bla extension:

location / {
    methods GET POST PUT;
    cgi_pass .bla /root/webserv/cgi_tester;
}

14. Multiple Ports / Virtual Hosts

Multiple server blocks can listen on different ports, each serving different content:

server {
    listen 0.0.0.0:8080;
    server_name main.localhost;
    root ./www;
    # ...full website...
}

server {
    listen 0.0.0.0:8081;
    server_name second.localhost;
    root ./www;
    client_max_body_size 1M;
    # ...different config...
}

Server selection: matched by Host header + port. If no match, the first server on that port is the default.

15. Body Size Limits

Body size limits can be set at three levels (most specific wins):

  1. http { client_max_body_size 5M; } — Global default
  2. server { client_max_body_size 10M; } — Per-server override
  3. location /upload { client_max_body_size 20M; } — Per-location override

When exceeded → 413 Payload Too Large

# Test body size limit (11MB to a 10M limit)
dd if=/dev/urandom bs=1M count=11 2>/dev/null | \
  curl -X POST --data-binary @- http://localhost:8080/upload
# Response: 413 Payload Too Large

16. Chunked Transfer Encoding

Full support for Transfer-Encoding: chunked in both directions:

  • Incoming: Chunked requests are decoded incrementally and streamed to CGI or buffered for handlers
  • CGI streaming: Large POST bodies are streamed chunk-by-chunk to CGI stdin via pipes (supports gigabyte-scale uploads)
  • Flow control: Back-pressure when CGI buffer exceeds threshold, client read paused until CGI catches up
# Send chunked request
curl -X POST -H "Transfer-Encoding: chunked" \
  -d "hello world" http://localhost:8080/cgi-bin/database.py

# Large chunked upload to CGI
python3 -c "
import http.client
conn = http.client.HTTPConnection('localhost', 8080)
conn.request('POST', '/cgi-bin/database.py',
  body='username=chunked_user&password=test',
  headers={'Transfer-Encoding': 'chunked', 'Content-Type': 'application/x-www-form-urlencoded'})
print(conn.getresponse().read().decode())
"

17. Non-blocking & Resilience

Guaranteed behaviors:

  • Server never blocks — single poll() drives everything
  • Server never crashes — all exceptions caught, all memory managed
  • Client disconnections handled gracefully (POLLHUP/POLLERR)
  • CGI timeouts: kill SIGTERM → SIGKILL escalation
  • Client timeouts: 60-second inactivity → connection closed
  • No hanging requests — poll timeout ensures regular timeout checks
  • Stress tested: 128 concurrent workers × 50 requests each
